The Problem¶

The ultimate subway dilemma. You arrive at a station that has both local and express trains. A local train just arrived. Should you hop on or wait for the express train?

To solve this problem, we are going to simplify it greatly by only optimizing for time between express stations. There are several factors that one considers beyond just travel time (e.g. having a seat, over-crowding, smells), but those will be discarded for this analysis.

Importing and formatting data¶

Raw data: http://web.mta.info/developers/data/nyct/historic/A%20Division%20Historical%20Data%20May%202011.zip

Documentation: http://web.mta.info/developers/data/nyct/historic/A%20Division%20Historical%20Data%20Documentation.txt

GTFS schedule data: http://web.mta.info/developers/data/nyct/subway/google_transit.zip

The raw data comes as a collection of data from each day. These need to be combined into a single pandas DataFrame, which we will call 'data.' It comes with a timestamp but needs to be converted to datetime and set as the index.

Each train trip has a unique id that are reused each day, so to get the service number, we need to combine the service date with the train id to get the service number - a unique identifier for a single trip on a single day.

Finally, we import the stops from the GTFS schedule data to merge with 'data' and get stop names.

The raw data contains 3.6+ million events: both arrivals and departures at every stop for all lines that are collected. An average rider only cares about the time from one train arriving until arriving at the next station. The time between arrival and departure is just additional wait time for the rider. So, all departure times are discarded.

Furthermore, we need to separate the data into the appropriate line: red (1,2,3) and green (4,5,6).

Processing the data¶

Now that the data is loaded, it needs to be processed for each line. Riders do not care what time a train comes. They care about how long it takes between stops. The data is processed to collect the difference in timestamps between stop.

The function below discards everything but the express stops for the selected line. Then the data is transformed for the relevant data for each service (train_id/date combo).

Please note, that an artificial cutoff of 15 minutes per station is included to remove extended delays from rare events.

def dx_color(train,color):
    unique = train['service'].unique()
    line = train['route_id'].values[0]
    if color  in 'red':
        usestops = ['Chambers St', '14 St', '34 St - Penn Station', 'Times Sq - 42 St', '72 St', '96 St']

    expstop = []
    for x in train['stop_name']:
        if x in usestops:
            expstop.append(True)
        else:
            expstop.append(False)

    train = train[expstop].sort(['service'])


    start = []
    stop = []
    startnum = []
    stopnum = []
    duration = []

    stoplon = []
    weekday = []
    hour = []

    service = []

    second = []
    for x in unique:
        df = train[ train[ 'service' ] == x] 
        if len(df) > 1:
            df = df.sort_index()
            direct = df['direction_id'].values
            trainstops = df['stop_name'].values
            times = df.index
            services = df['service'].values

                            route.append(df['route_id'].values[t])
                            service.append(services[t])
                            train_id.append(df['train_id'].values[t])
                            minute.append(times[t].minute)
                            second.append(times[t].second)

    test = pd.DataFrame({

The Model¶

I used an Extra Random Forest Regressor with bootstrapping as the main component of my model. Using a grid search, the optimum parameters were found.

Because all of the input is categorical data (route, starting station, ending station, day of the week, and hour), everything went through a One Hot Encoder before the extra random forest.

Implementing the model in Flask¶

The web app first scraps the MTA status page to make sure there is "Good Service." If not, the user is informed about what kind of service disruption is occurring and to check the MTA status page for more information.

Assuming good service, the subway line, direction, and applicable model are determined from the starting and stopping stations. The user is informed if these inputs are either the same station or not on the same line.

The number of stops traveled is calculated and the sum of the time to travel between each station at that moment is calculated. The difference is returned to the user with the name of the fastest train.

What's next and limitations?¶

Weather data, specifically temperature and current precipitation need to be explored to see if that is a factor. The weather for the entire month of May 2011 will need to be matched to the corresponding event times in the data. However, there are enduser implications of this. If this was developed as a mobile app, its current form would work offline. Collecting current weather data would require an internet connection, which might not be in the user's best interest.

Current real-time testing of the models using the MTA's live data feed to further validate the model.

The user interface is confusing and not as beautiful or intuitive as it can be.
